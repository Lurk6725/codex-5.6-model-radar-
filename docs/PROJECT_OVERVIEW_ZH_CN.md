# GPT‑5.6 Codex 模型雷达：中文项目说明

## 项目定位

本项目是一个独立、非商业、开源的 GPT‑5.6 Codex 模型长期监控与二次分析项目。它以 Codex Radar 公开展示的测试结果为基础，持续记录 Luna、Terra、Sol 各模型家族及不同推理强度的表现，并生成可复现的统计分析和使用建议。

本项目不是 OpenAI、Codex Radar 或 DeepSWE 的官方项目，也不代表这些组织的立场。

## 数据来源

主要数据来源为：

- Codex Radar 公开网页；
- Codex Radar 公开摘要；
- 人工核对的公开页面快照；
- 本仓库根据公开结果计算的衍生指标。

来源数据应保留：

- 测试时间；
- 批次 ID；
- 模型与推理强度；
- 原始通过数；
- 逐题结果（公开可得时）；
- Token、执行时间和估算费用；
- 来源 URL；
- 数据来源说明；
- 四舍五入、部分重跑或字段缺失等不确定性。

项目不使用私有凭证，不规避访问限制，也不冒充 Codex Radar 官方镜像。未来如需私有 API、完整历史导出或受保护数据集再分发，将先与来源维护者沟通。

## 项目为什么存在

固定十题的单轮原始通过数容易被随机 Agent 轨迹影响，也无法区分简单题和困难题。本项目试图回答以下更长期的问题：

1. 哪个模型档位在多轮测试中最稳定？
2. 模型推理强度提高是否真的带来单调收益？
3. 相同订阅额度下，哪个档位提供更高的有效能力？
4. 哪些批次可能受后台配置、Harness、自动 Review 或服务状态异常影响？
5. 不同工作类型应该优先使用哪个模型？

## 难度加权

每道任务根据历史通过率获得一个软逆通过率权重：

```text
原始难度_i = 1 / sqrt(历史通过率_i)
题目权重_i = 100 × 原始难度_i / sum(原始难度)
难度加权分 = sum(题目权重_i × 是否通过_i)
```

使用平方根是为了避免最难的一道题完全支配排行榜。项目还保留原始通过数，方便同时观察绝对成功数和任务难度结构。

## 主要统计指标

- **原始通过数**：十题中完成的数量；
- **难度加权分**：0–100；
- **加权 IQ**：难度加权分乘以 1.5；
- **额度效率**：难度加权分除以来源网站估算费用；
- **滚动中位数**：最近三轮或五轮有效批次的中位数；
- **滚动均值**：观察长期中心趋势；
- **波动范围**：近期最好与最差表现；
- **Pareto 前沿**：没有其他配置同时做到更便宜且更强的模型档位；
- **异常修正趋势**：异常批次保留但降低长期权重。

## 异常批次

异常批次不会被删除。项目会完整保留它们，因为异常可能反映真实服务状态变化。

默认异常信号包括：

- 全体通过数相较上一轮下降至少 15%；
- 同时总费用或总 Token 上升至少 30%；
- 多个互不相关的档位同时出现“更贵、更慢、成功更少”；
- 来源页面标记了 Harness、上下文、自动 Review 或后台配置变化。

异常批次应在最新报告中明确标记，并在长期趋势中使用较低可靠度权重。

## 订阅额度解释

本项目面向 Codex 订阅用户时，不将 API 的长上下文阶梯价格直接套用到订阅额度上。

网站的美元等价费用用于相对比较，但不是精确账户账单。总 Token 和耗时用于判断 Agent 膨胀、上下文扩张和交互延迟，而不是直接代替额度消耗。

## 推荐原则

项目不会因为单轮测试结果就更换默认推荐。通常需要至少满足以下一种证据标准：

- 连续三轮有效批次出现一致变化；
- 最近五轮中位数发生实质变化；
- 一个档位持续处于成本—能力 Pareto 前沿；
- 来源或官方确认了模型/Harness 配置变化；
- 在可复现的私人项目测试中持续获得明显优势。

当前推荐通常分为：

- 机械、低风险、允许重试；
- 日常开发默认；
- 高难、额度优先；
- 高难、最高能力优先；
- 当前不建议作为默认的档位。

## 双语维护要求

每次新增测试数据并完成分析后，必须同时更新：

- 中文完整报告；
- 英文完整报告；
- 中英文合并入口；
- 中英文推荐摘要；
- 如有方法变化，中英文方法说明。

中英文内容应表达相同结论、相同数据和相同置信度。若翻译尚未完成，不应把单语言报告标记为正式最新版本。

## 当前限制

- 测试集仅有十道题；
- 每个任务通常只运行一次；
- Agent 轨迹具有随机性；
- 任务难度权重可能包含模型家族偏好；
- 公开摘要可能缺失逐题历史或部分重跑信息；
- 网站估算费用并非官方订阅扣款；
- 固定代码任务不能代表所有真实开发场景。
